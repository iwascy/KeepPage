## 一、背景

KeepPage 当前已经具备一条可运行的 archive-first MVP 链路：

- 扩展负责抓取页面、生成本地归档、写入 IndexedDB 队列，并继续同步到 API。
- API 负责归档初始化、对象上传、完成入库、书签检索与详情读取。
- Web 负责列表、搜索、详情、版本切换和归档预览。

这条链路适合常规网页收藏，但并不适合下列敏感内容：

- 含账号信息、工单内容、客户资料、支付信息的后台页面。
- 仅限个人查看的知识库、私有文档、草稿或研究页面。
- 用户明确不希望以明文进入服务端存储、服务端搜索和 Web 预览的数据。

当前产品的隐私能力主要停留在“按账号隔离”，还没有“按敏感级别隔离”。对于敏感网页，用户需要的是：

1. 本地数据不能明文长期落盘。
2. 服务端不能直接看到标题、URL、正文和质量诊断。
3. 查看私密内容时需要二次解锁，而不是只靠账号登录。
4. 用户可以明确区分“普通保存”和“私密保存”。

因此，需要为 KeepPage 增加一套独立且可落地的私密模式。

## 二、产品目标

### 2.1 目标

1. 让用户可以将敏感网页保存到独立的私密库，而不是混入普通归档。
2. 让私密内容在本地和云端都默认以加密形式保存。
3. 让私密内容默认不参与普通搜索、普通列表、普通详情和普通预览。
4. 保持 KeepPage “本地先成功，再异步同步”的产品原则。
5. 在可控范围内支持后续跨设备同步和恢复能力。

### 2.2 非目标

本期不追求：

- 服务端可见明文的私密全文检索。
- 私密归档公开分享链接。
- 团队协作共享私密库。
- 复杂权限模型，例如库内多角色和细粒度协作者权限。
- 对遗留普通归档的自动批量迁移。

## 三、设计原则

1. **真实私密，不做伪开关。**
   私密模式不能只是 UI 隐藏或加一个 `is_private` 标签，必须覆盖本地存储、同步链路、服务端存储和 Web 展示。
2. **默认最小暴露。**
   锁定状态下，只暴露最小必要信息，例如条目数量、同步状态、更新时间，不暴露标题、URL、域名、对象键和正文。
3. **本地优先，解锁在端侧。**
   私密内容的解密发生在用户设备端，服务端只存加密内容和最小元数据。
4. **先做可用，再做完整。**
   先落地本机私密，再逐步扩展到跨设备同步、恢复和迁移。
5. **风险明确告知。**
   忘记私密口令或恢复码后，系统无法帮助恢复明文内容，这一点必须在产品中明确提示。

## 四、目标用户与典型场景

### 4.1 目标用户

- 使用 KeepPage 保存工作资料的重度用户。
- 需要保存后台页面、工单系统、财务系统、CRM、运营平台的用户。
- 有本地知识管理需求，但对云端明文存储敏感的用户。

### 4.2 典型场景

#### 场景 A：保存管理后台页面

用户在公司后台系统中查看一条订单详情，希望留存页面快照，但不希望标题、URL 和正文进入服务端明文存储。

#### 场景 B：保存个人私密资料

用户保存一篇仅供自己研究使用的内部文档，希望在本机长期保留，但不希望 Web 列表直接显示内容。

#### 场景 C：无痕窗口保存

用户在浏览器无痕窗口中打开页面时，更倾向于默认进入私密模式，降低误保存到普通库的风险。

## 五、核心定义

| 名词 | 定义 |
| --- | --- |
| 普通模式 | 沿用现有 KeepPage 链路，元数据与归档按普通书签方式保存 |
| 私密模式 | 将内容保存到独立私密库，内容默认加密，本地或云端都不以明文暴露 |
| 私密库 | 用户名下独立的加密归档空间，和普通书签空间分离 |
| 本机私密 | 私密内容只保存在当前浏览器设备，服务端不持有归档内容 |
| 同步私密 | 私密内容加密后同步到服务端，允许多设备查看 |
| 锁定 | 私密库未解锁的状态，仅能看到极少量概览信息 |
| 解锁 | 用户输入私密口令后，在当前设备短时持有解密能力的状态 |

## 六、方案总览

KeepPage 私密模式采用“双模式 + 独立私密库”的方案：

### 6.1 模式划分

#### 普通保存

- 适合大多数普通网页。
- 沿用现有抓取、同步、列表、详情和预览链路。

#### 私密保存

- 适合敏感网页。
- 默认写入私密库。
- 本地数据加密落盘。
- 服务端只接收最小必要信息和加密对象。
- Web 端需解锁后才能显示完整内容。

### 6.2 私密模式的两档能力

#### 第一档：本机私密

- 仅当前浏览器设备可见。
- 不上传归档主档到服务端。
- 适合首版落地，复杂度低，隐私最强。

#### 第二档：同步私密

- 归档主档、派生物和元数据加密后再同步。
- 其他设备登录后需要再次输入私密口令才能解锁。
- 适合作为第二阶段能力。

### 6.3 产品主张

KeepPage 对用户的承诺是：

- 普通模式提供最好用的 archive-first 收藏体验。
- 私密模式提供更强隐私保护，但会牺牲一部分搜索和分享能力。

## 七、用户流程

### 7.1 首次创建私密库

1. 用户在扩展中首次切换到“私密”保存模式。
2. 系统弹出引导，说明私密模式的隐私边界和代价。
3. 用户设置私密口令。
4. 系统生成恢复码并提示用户妥善保存。
5. 用户设置自动锁定时长，例如 5 分钟、15 分钟、1 小时、浏览器关闭后锁定。
6. 创建成功后，扩展进入“已启用私密库”状态。

### 7.2 保存当前页面为私密归档

1. 用户在扩展中选择“私密保存”。
2. 系统校验当前私密库是否已解锁。
3. 若未解锁，先提示输入私密口令。
4. 抓取成功后，归档主档、提取文本、截图、质量报告在本地加密写入。
5. 若为本机私密，则任务停留在本地私密队列。
6. 若为同步私密，则加密对象异步上传到服务端私密链路。
7. 用户看到“已私密保存”的成功提示，但默认不显示页面标题和 URL。

### 7.3 在 Web 端查看私密内容

1. 用户进入 Web 端“私密库”入口。
2. 若未解锁，只展示锁定态摘要。
3. 用户输入私密口令后，浏览器端获取解密所需材料并在本地完成解锁。
4. 列表、详情、预览、版本切换和下载在解锁态下可用。
5. 解锁超时或用户主动锁定后，页面退回锁定态。

### 7.4 自动锁定

出现以下任一情况时，系统自动锁定私密库：

- 达到自动锁定超时时间。
- 浏览器或标签页关闭。
- 用户手动点击“立即锁定”。
- 口令校验失败次数达到阈值后触发冷却。

### 7.5 普通与私密之间的迁移

1. 用户在已解锁设备上发起“转为私密”或“转为普通”。
2. 系统重新打包并走目标链路。
3. 原条目默认保留，待新条目成功后再提示用户是否删除旧条目。
4. 首版不提供批量迁移，只支持单条迁移。

## 八、信息架构

### 8.1 扩展侧

- 保存模式切换：`普通` / `私密`
- 私密库状态：`未启用` / `已锁定` / `已解锁`
- 私密任务列表：与普通任务列表分开展示
- 私密设置：
  - 自动锁定时长
  - 无痕窗口默认私密
  - 域名规则
  - 恢复码查看入口

### 8.2 Web 侧

- 普通工作台：保持现状
- 私密库入口：新增一级入口，不与普通列表混排
- 私密库页面：
  - 锁定态概览
  - 解锁态列表
  - 解锁态详情与版本页

### 8.3 服务端

- 普通链路：`/captures/*`、`/bookmarks/*`、`/objects/*`
- 私密链路：独立路由与独立存储域，不与普通链路复用

## 九、功能需求

### 9.1 扩展侧功能需求

#### 9.1.1 保存模式切换

- 在 Side Panel 顶部增加“保存模式”选择器。
- 选择器需持久化最近一次选择。
- 无痕窗口默认预选“私密”。
- 域名规则命中时，自动切为“私密”，但允许用户手动改回。

#### 9.1.2 私密库启用与解锁

- 首次启用时引导用户创建私密口令。
- 解锁前不能进行私密保存。
- 口令仅在当前会话内保存在内存。
- 支持手动锁定。

#### 9.1.3 本地加密存储

- 私密任务的 `archiveHtml`、`extractedText`、截图、缩略图、质量报告必须加密后再写入 IndexedDB。
- 明文任务壳仅保留：
  - `id`
  - `status`
  - `createdAt`
  - `updatedAt`
  - `owner`
  - `isPrivate`
  - `privateMode`
  - `syncState`
- 本地日志不记录标题、URL、对象键、正文片段。

#### 9.1.4 私密任务视图

- 提供私密任务列表和状态摘要。
- 锁定状态下，任务卡片不展示标题、域名、质量详情。
- 解锁状态下可查看：
  - 标题
  - 保存时间
  - 版本数
  - 同步状态
  - 质量提示

#### 9.1.5 重试与恢复

- 本机私密任务失败时，允许原地重试，不要求重新抓取。
- 同步私密上传失败时，保留加密本地副本，允许继续同步。
- 口令错误不应破坏已存在的本地加密数据。

### 9.2 Web 端功能需求

#### 9.2.1 私密库锁定态

- 显示：
  - 私密条目总数
  - 最近更新时间
  - 待同步数量
  - 是否已启用同步私密
- 不显示：
  - 标题
  - URL
  - 域名
  - 标签
  - 质量原因
  - 对象键

#### 9.2.2 私密库解锁态

- 支持查看私密列表、详情、版本和预览。
- 解锁仅对当前标签页或当前浏览器会话生效。
- 用户主动刷新或锁定后，需要重新解锁。

#### 9.2.3 搜索与筛选

- 锁定态不提供内容搜索。
- 解锁态支持本地搜索标题、标签和域名。
- 首版不支持服务端私密全文检索。

#### 9.2.4 下载与导出

- 解锁后允许下载私密归档。
- 下载动作需要明确提示“文件将以明文保存到本地系统下载目录”。

### 9.3 服务端功能需求

#### 9.3.1 独立私密 API

新增私密专用路由，不复用普通 `bookmarks` 与 `objects`：

- 私密库初始化与状态查询
- 私密对象上传
- 私密归档完成写入
- 私密列表最小元数据查询
- 私密加密对象读取

#### 9.3.2 最小明文原则

服务端可以知道：

- 条目归属用户
- 条目创建时间
- 是否同步成功
- 对象大小
- 版本数量

服务端不应看到：

- 标题明文
- URL 明文
- 正文明文
- 提取文本明文
- 质量原因明文
- 预览 HTML 明文

#### 9.3.3 去重策略

- 普通模式现有的 `URL + htmlSha256` 去重不适合直接照搬到私密模式。
- 私密模式改为客户端基于私密库密钥计算稳定去重指纹。
- 去重范围仅限同一私密库内部。

### 9.4 设置与规则需求

- 支持开关“无痕窗口默认私密”。
- 支持域名级规则：
  - 总是私密
  - 默认普通
  - 继承全局设置
- 支持自动锁定时长配置。
- 支持查看恢复码提示页，但不支持明文长期展示恢复码。

## 十、隐私与安全要求

### 10.1 加密要求

- 本地与同步私密都必须使用强加密。
- 建议使用：
  - 口令派生：Argon2id
  - 内容加密：AES-GCM
  - 去重指纹：HMAC-SHA256

### 10.2 密钥要求

- 私密口令不上传服务端。
- 主密钥不落磁盘明文。
- 每条归档使用独立数据密钥。
- 数据密钥由私密库主密钥封装。

### 10.3 日志与调试要求

- 私密模式下默认关闭详细调试日志。
- 错误日志不得包含：
  - 页面标题
  - 页面 URL
  - 归档正文
  - 对象键
  - 提取文本片段

### 10.4 会话安全要求

- 私密库解锁状态独立于账号登录状态。
- 账号登录成功不代表私密库自动解锁。
- 自动锁定后必须重新输入私密口令。

## 十二、版本规划

### 12.1 V1：本机私密

#### 范围

- 扩展支持创建私密库。
- 私密任务加密后写入本地 IndexedDB。
- 支持锁定、解锁、自动锁定、域名规则、无痕窗口默认私密。
- 不做跨设备同步。
- Web 端仅展示“当前账号已启用私密能力”，不展示私密内容。

#### 目标

先交付“真实可用的本地私密能力”，验证用户是否会稳定使用。

### 12.2 V2：同步私密

#### 范围

- 服务端新增私密对象上传与最小元数据存储。
- Web 端支持私密库锁定态与解锁态。
- 私密内容加密同步到服务端。
- 支持跨设备查看。

#### 目标

把私密模式从“单设备能力”推进为“跨设备能力”。

### 12.3 V3：迁移、恢复与增强搜索

#### 范围

- 单条普通转私密 / 私密转普通。
- 恢复流程优化。
- 本地解锁态搜索增强。
- 私密库导出与备份。

## 十三、里程碑与验收

### 13.1 V1 验收标准

1. 用户可以在扩展中创建私密库并成功解锁。
2. 私密任务在本地 IndexedDB 中不以明文保存 `archiveHtml` 和 `extractedText`。
3. 锁定状态下，任务列表不显示标题、域名和质量详情。
4. 自动锁定生效，重新查看私密任务需要再次输入口令。
5. 无痕窗口默认命中私密模式。

### 13.2 V2 验收标准

1. 私密归档可以加密后同步到服务端。
2. 服务端数据库中不出现私密条目的标题、URL 和正文明文。
3. Web 端支持锁定态与解锁态切换。
4. 解锁后可查看至少一个真实私密归档版本。
5. 服务端普通搜索接口不会返回私密内容。

## 十四、成功指标

### 14.1 核心指标

- 私密模式开启率
- 私密保存成功率
- 私密任务重试恢复率
- 私密库解锁成功率
- 私密模式 7 日留存

### 14.2 安全指标

- 锁定态信息泄露事故数为 0
- 服务端明文私密内容写入事故数为 0
- 私密日志脱敏失败事故数为 0

## 十五、风险与权衡

### 15.1 用户教育成本

私密口令、恢复码、自动锁定等概念会增加学习成本，首屏引导必须足够清楚。

### 15.2 能力牺牲

私密模式天然会牺牲以下能力：

- 服务端全文搜索
- 即开即看的 Web 预览
- 部分跨设备无感体验

### 15.3 实现复杂度

如果直接在当前普通链路上打补丁，容易产生：

- 普通 API 误返回私密条目
- 普通日志误记录私密内容
- 普通对象存储误读私密对象

因此必须坚持“独立私密库 + 独立私密链路”的方案。

## 十六、开放问题

1. V1 是否要在扩展侧提供“私密任务导出备份”能力。
2. V2 是否允许同一账号下存在多个私密库。
3. 私密模式下是否保留截图生成，还是默认关闭以减少敏感图片落盘。
4. 私密模式下 SingleFile 失败时，是否允许使用当前 DOM fallback，还是直接失败更符合隐私预期。
5. Web 端私密库解锁状态应按“标签页级”还是“浏览器会话级”管理。

## 十七、建议执行顺序

1. 先做 V1，本机私密闭环。
2. 在 V1 落地后补一轮真实浏览器联调与本地存储检查。
3. 再推进 V2，同步私密链路与 Web 解锁态。
4. 最后做迁移、恢复和增强搜索。

## 十八、结论

KeepPage 私密模式不应被定义成一个普通的“隐藏内容”功能，而应被定义成一个独立的私密库能力。它的目标不是让敏感条目“看起来不明显”，而是让敏感条目在扩展、本地存储、同步链路、服务端和 Web 查看路径上都具备明确的隐私边界。

从落地路径看，建议优先交付“本机私密”，尽快验证用户对真实私密保存的需求，再逐步扩展到“同步私密”与跨设备查看。
