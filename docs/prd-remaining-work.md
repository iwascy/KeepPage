# [私密模式 PRD] KeepPage 剩余工作清单

## 背景

当前仓库已经具备一条可运行的 archive-first MVP 链路：

- Chrome MV3 扩展已接入 SingleFile，并能完成本地归档、质量评估、队列持久化和基础同步。
- API 已支持 `memory` / `postgres` 双存储，并跑通 `captures/init`、`captures/complete`、`bookmarks` 列表 / 详情与对象读取。
- Web 端已支持归档列表、搜索、质量筛选、基础统计，以及归档查看页、版本切换和 iframe 预览。

目前最大的缺口，不在“有没有页面”，而在“归档主档是否真正闭环落库”和“真实使用体验是否经过浏览器验证”。

## P0：真实浏览器联调

### 目标

在真实 Chrome 环境中验证当前扩展链路，而不是只依赖类型检查和构建成功。

### 具体工作

1. 加载扩展构建产物到 Chrome。
2. 在以下页面类型上分别执行一次保存：
   - 普通静态文章页
   - 含延迟图片的页面
   - 含 iframe 的页面
   - SPA 页面
   - 明显包含 canvas 或 video 的页面
3. 验证 Side Panel 状态机是否按预期变化：
   - `queued -> capturing -> validating -> local_ready -> upload_pending -> uploading -> synced`
4. 验证本地预览是否能直接打开归档 HTML。
5. 验证失败后的两类动作：
   - 继续同步已有本地归档
   - 按当前 Profile 重新抓取原始 URL
6. 记录至少一轮失败样例和原因，沉淀为后续规则输入。

### 验收标准

- 能在 Chrome 内稳定完成至少 5 类页面的手工验证。
- 至少一条任务能从本地归档成功走到 `synced`。
- 至少一条失败任务能通过“继续同步”或“重新抓取”恢复。
- 没有发现阻断性 MV3 注入、消息通信或权限问题。

## P1：对象存储直传闭环

### 目标

让 `archive.html` 真正上传到对象存储，而不是只在 API 和本地队列之间做“元数据同步”。

### 当前状态

进行中。开发态本地对象存储闭环已经完成，真实云对象存储预签名上传尚未接入。

### 具体工作

1. 当前已完成：
   - `POST /captures/init` 返回真实可上传地址
   - 扩展侧在 `init` 后直传 `archive.html`
   - `POST /captures/complete` 基于真实对象存在完成版本写入
2. 继续推进真实对象存储实现。
   - 本地开发优先接 MinIO 或兼容 S3 的方案。
   - 线上再切到 S3 / R2 / OSS。
3. 明确对象键命名规则，至少覆盖：
   - `archive.html`
   - `meta.json`
   - 后续可扩展 `thumb.webp`、`screenshot.webp`
4. 补充失败重试策略：
   - 上传失败保留本地队列
   - 不重复生成本地归档
   - 同一 `htmlSha256` 可安全重试上传

### 验收标准

- 一次保存后，能在对象存储中看到真实 `archive.html` 对象。
- API 中记录的 `html_object_key` 与实际对象一致。
- 断网或服务异常时，任务会停在 `upload_pending`，恢复后可继续上传。
- 切换到真实 S3 兼容存储后，不需要改扩展侧上传协议。

## P1：队列与失败恢复体验

### 目标

把当前可用的 Side Panel，提升为真正可处理待办任务的保存队列。

### 具体工作

1. 为任务列表增加待处理优先视图。
   - 默认优先显示 `upload_pending` 和 `failed`
   - 再显示最近成功记录
2. 将“继续同步”和“重新抓取”拆成两个独立动作。
3. 在失败详情中加入明确 CTA：
   - 测试连接
   - 继续同步
   - 重新抓取
   - 打开设置
4. 增加队列摘要信息。
   - 待同步数量
   - 失败数量
   - 最近同步成功时间
5. 对低质量任务增加更直接的提示。
   - 当前质量等级
   - 核心诊断原因
   - 推荐重试的 capture profile

### 验收标准

- 用户能在 Side Panel 中一眼看到当前待处理任务。
- 同步失败时，不需要猜测下一步操作。
- 低质量归档可以被快速识别并触发重试。

## P2：域名级 capture 规则

### 目标

把 KeepPage 从“能保存”推进到“会保存”。

### 具体工作

1. 增加域名级 capture profile 配置。
2. 引入站点规则层，支持：
   - 自动展开正文
   - 关闭 cookie banner
   - 展开评论或代码块
   - 等待延迟内容加载
3. 预留 user script 机制。
4. 为已知高风险站点记录限制说明：
   - iframe 风险
   - canvas 风险
   - video 风险
   - 浏览器受限页面

### 验收标准

- 同一域名可自动匹配指定 profile。
- 至少支持 2 到 3 个典型站点规则。
- 质量评分能体现规则执行前后的差异。

## P2：增量同步与搜索增强

### 目标

为多端同步和数据量增长提前打基础。

### 具体工作

1. 设计并实现 `sync_cursor` / `sync_op`。
2. 客户端支持增量拉取元数据变更。
3. HTML 保持按需下载，不做默认全量下发。
4. 搜索从当前 API 查询进一步增强：
   - 排序策略优化
   - 更好的正文检索
   - 为后续专用搜索引擎接入预留接口

### 验收标准

- 同一用户的多端可以基于 cursor 拉取变更。
- 不需要全量重拉书签数据。
- 搜索字段和排序规则有明确约定。

## 建议执行顺序

1. 先做真实浏览器联调，确认当前扩展链路在 Chrome 内没有阻断性问题。
2. 再做对象存储直传闭环，补齐 `archive.html` 主档真正落库。
3. 然后补 Side Panel 队列与失败恢复体验，让产品可日常使用。
4. 继续增强 Web 查看页与 Side Panel 体验，形成“保存后可查、可看、可验证”的完整闭环。
5. 最后再推进域名级规则、多端同步和搜索增强。
