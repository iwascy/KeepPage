## 一、背景

KeepPage 当前已经具备一条可运行的 archive-first 链路：

- 浏览器扩展可以抓取当前网页，生成本地归档并异步同步到后端。
- API 已支持书签列表、详情、对象读取、归档初始化和归档完成。
- Web 管理端已支持浏览、搜索、筛选和归档查看。

但对新用户来说，当前产品仍存在一个明显门槛：

- 用户已有大量浏览器书签时，无法一次性迁入 KeepPage。
- 用户手上已有 URL 清单、CSV、TXT、知识库链接集合时，无法批量导入。
- “导入书签”和“完整归档网页”是两种不同复杂度的动作，但当前产品没有把它们拆开。

这会直接影响新用户迁移效率，也会降低 KeepPage 作为个人知识归档入口的吸引力。

因此，需要设计一套完整的“批量导入”产品能力，让用户可以先把资产迁进来，再按需完成网页归档。

---

## 二、问题定义

当前用户在批量迁移收藏时，会遇到以下问题：

1. 无法从浏览器已有书签直接导入。
2. 无法从 URL 列表、CSV、TXT 等来源批量创建书签。
3. 无法在导入前预览重复、无效链接、目标文件夹与标签映射。
4. 无法批量将导入条目进一步转成完整网页归档。
5. 无法查看导入结果、失败原因、去重结果和后续归档进度。

本 PRD 解决的核心问题，不是“怎么一次性塞很多链接进系统”，而是：

**如何让 KeepPage 以低门槛接收用户原有收藏资产，并逐步转化为高质量的可搜索、可归档、可管理的长期知识资产。**

---

## 三、产品目标

### 3.1 目标

1. 支持浏览器书签的批量导入。
2. 支持 URL 清单的批量导入。
3. 导入前支持预检查、预览和去重提示。
4. 导入后支持按需归档，且与现有 archive-first 链路一致。
5. 支持查看导入任务进度、结果、失败原因和重试。
6. 支持保留来源信息，如原文件夹路径、原始标题、导入来源。

### 3.2 非目标

本期不做以下内容：

- 不做服务端网页抓取作为主方案。
- 不做自动登录、验证码处理、反爬绕过。
- 不做跨账号共享导入任务。
- 不做复杂的智能分类或 AI 自动打标签。
- 不做浏览器历史记录导入。

---

## 四、核心设计原则

### 4.1 双模式导入

批量导入必须明确拆成两种模式：

- **轻导入**：仅导入链接和元数据，立即生成书签记录。
- **归档导入**：在轻导入基础上，逐条调用浏览器扩展完成网页归档。

### 4.2 先入库，再归档

导入与归档不强绑定。用户应当可以：

- 先把全部链接导入 KeepPage。
- 再选择其中一部分进行归档。
- 或者让系统后台逐步归档。

### 4.3 复用现有归档链路

批量归档必须复用现有扩展抓取、质量评估、上传、版本管理和去重逻辑，不能为导入场景单独再造一套网页保存流程。

### 4.4 结果透明

用户必须清楚看到：

- 哪些条目成功导入。
- 哪些条目被合并或跳过。
- 哪些条目归档失败。
- 失败原因是什么。
- 下一步可执行什么操作。

## 五、目标用户

### 5.1 新迁移用户

特征：

- 已在 Chrome、Edge、Firefox、Safari 中积累大量书签。
- 希望快速迁移到 KeepPage。
- 不要求每个网页都立即完整归档。

### 5.2 重度资料收藏用户

特征：

- 常维护论文、博客、教程、工具站、文档站链接集合。
- 经常从 Notion、飞书、Markdown、CSV 中整理链接。
- 需要批量沉淀资料库。

### 5.3 归档重视型用户

特征：

- 不只是保存链接，更重视页面可离线查看与长期保留。
- 愿意在导入后让系统逐步做归档。

## 六、典型使用场景

### 6.1 从浏览器迁移已有书签

用户第一次使用 KeepPage，希望把 Chrome 里的几百个书签导入进来，保留原有文件夹结构，并先完成轻导入。

### 6.2 从文档中导入链接清单

用户从 Markdown、TXT 或 CSV 中整理了一批资料链接，希望一次性导入系统，后续再筛选重点资料做完整归档。

### 6.3 批量归档一个专题集合

用户导入了一个“AI 产品设计资料”文件夹，希望先导入全部链接，再对其中前 50 条自动完成网页归档。

### 6.4 补归档

用户之前只导入了元数据，现在希望按文件夹、标签或域名批量补做归档。

## 七、产品范围

### 7.1 输入来源

首期支持以下导入来源：

1. 浏览器扩展读取浏览器书签树。
2. 浏览器导出的书签 HTML 文件。
3. 用户粘贴多行 URL。
4. 上传 `.csv`、`.txt`、`.md` 文件并提取 URL。

后续可扩展：

- 当前打开标签页批量导入。
- 标签页组导入。
- 从第三方服务导入。

### 7.2 导入模式

首期支持三种模式：

1. 仅导入链接。
2. 导入后排队归档。
3. 导入并立即开始归档。

### 7.3 输出结果

导入完成后，系统会产出：

- 导入任务记录。
- 导入条目明细。
- 新创建或合并后的书签。
- 可选的归档任务与归档版本。

## 八、方案概述

### 8.1 核心流程

1. 用户选择导入来源。
2. 系统解析原始输入。
3. 系统做 URL 清洗、去重预检和结构预览。
4. 用户确认导入规则。
5. 系统执行轻导入。
6. 如果选择归档模式，则把可归档条目交给扩展队列处理。
7. 用户在导入详情页查看最终结果与后续归档进度。

### 8.2 首期推荐策略

首期默认推荐用户选择“仅导入链接”。

原因：

- 上手成本最低。
- 失败率更低。
- 更适合大批量迁移。
- 用户可以先建立完整收藏体系，再对重点内容做高成本归档。

对于已安装扩展且明确希望保存网页内容的用户，再引导其选择“导入后排队归档”。

## 十、详细功能需求

### 10.1 来源选择

系统应支持以下来源卡片：

- 从浏览器导入
- 导入书签 HTML
- 粘贴 URL 列表
- 上传 CSV 或 TXT

每种来源卡片都需要说明：

- 适合什么场景
- 是否保留文件夹结构
- 是否依赖扩展
- 是否支持后续批量归档

---

### 10.2 输入解析

#### 功能要求

1. 识别合法 URL。
2. 自动去掉空行、非法行、无协议行中的噪声。
3. 对 URL 做规范化处理。
4. 提取标题、文件夹路径、标签等元数据。
5. 对书签 HTML 解析为树状结构。

#### 非法输入处理

以下内容不应进入正式导入：

- 空字符串
- 无法解析的 URL
- `javascript:` 链接
- `data:` 链接
- `chrome://`
- `about:`

处理策略：

- 在预览中标记为“不可导入”
- 显示原因
- 默认不计入执行列表

---

### 10.3 预检查与预览

#### 预览页必须展示

- 原始总条目数
- 有效条目数
- 无效条目数
- 文件内重复数
- 与站内现有书签重复数
- 预计新建书签数
- 预计合并书签数
- 预计跳过条目数

#### 结构化预览

用户应能看到：

- 前若干条样例
- 文件夹结构预览
- 域名分布
- 来源类型

#### 站内重复提示

如果某个 URL 在站内已存在，预览中应明确标识：

- 已存在书签
- 已存在归档版本
- 已存在相同内容版本

---

### 10.4 导入配置

#### 基础配置

- 导入模式：仅导入链接 / 导入后排队归档 / 导入并立即开始归档
- 目标位置：保留原文件夹结构 / 导入到指定文件夹 / 扁平化导入
- 标签策略：保留来源标签 / 追加导入批次标签 / 不追加标签
- 标题策略：优先原书签标题 / 优先网页真实标题 / 导入后再更新

#### 去重策略

默认策略：

- 以规范化 URL 为准做书签级去重。
- 命中现有书签时默认合并。
- 命中现有相同归档版本时默认跳过。

可选策略：

- 重复时跳过
- 重复时更新元数据
- 重复时仍保留为独立导入项记录

#### 归档配置

当用户选择归档模式时，应展示：

- 归档 Profile
- 并发度
- 失败后重试次数
- 域名限速
- 是否只归档前 N 条

首期建议默认：

- 并发度为 1
- 默认重试 1 次
- 每域名串行处理

---

### 10.5 执行导入

#### 轻导入执行逻辑

1. 为每个有效条目创建导入记录。
2. 按规范化 URL 与已有书签比对。
3. 决定是新建书签、合并、还是跳过。
4. 保存来源信息、文件夹路径、原始标题、导入批次。
5. 更新导入任务统计信息。

#### 归档导入执行逻辑

1. 轻导入成功后，筛选出可归档条目。
2. 将条目提交给扩展侧批量归档调度器。
3. 调度器逐个抓取网页并复用现有归档流程。
4. 将结果回写到导入条目与书签版本。

---

### 10.6 导入结果与详情

#### 导入任务概览

应展示：

- 任务名称
- 导入来源
- 创建时间
- 导入模式
- 总条目数
- 成功数
- 合并数
- 跳过数
- 失败数
- 归档成功数
- 归档失败数

#### 条目列表

每一条导入项应展示：

- 标题
- URL
- 域名
- 原文件夹路径
- 当前状态
- 去重结果
- 关联书签
- 是否已有归档
- 错误原因

#### 可执行操作

- 重试失败项
- 仅对未归档项继续归档
- 按域名筛选失败项
- 导出失败清单

## 十一、状态设计

### 11.1 导入任务状态

- `draft`
- `parsing`
- `ready`
- `running`
- `paused`
- `completed`
- `partial_failed`
- `failed`
- `cancelled`

### 11.2 导入条目状态

- `pending`
- `deduplicated`
- `created_bookmark`
- `queued_for_archive`
- `archiving`
- `archived`
- `skipped`
- `failed`

### 11.3 状态说明

#### `deduplicated`

表示命中已有书签或已有相同归档版本，本次未重复创建新内容。

#### `created_bookmark`

表示轻导入已成功，书签已入库，但还未开始归档。

#### `queued_for_archive`

表示已进入扩展批量归档队列，等待抓取。

#### `archived`

表示已成功生成或复用归档版本。

## 十二、去重与合并策略

### 12.1 去重维度

系统需要分两层去重：

1. 文件内去重
2. 站内去重

### 12.2 书签级去重

以规范化 URL 为主键语义。

如果命中已有书签：

- 默认不新建第二个书签。
- 把本次导入记录挂到已有书签上。
- 可更新标题、文件夹、标签等元数据。

### 12.3 归档级去重

如果进入归档流程：

- 相同 URL 且相同 HTML 指纹：视为完全重复，跳过。
- 相同 URL 但 HTML 不同：作为新版本保存。

### 12.4 合并原则

首期默认使用以下合并原则：

- 标题优先保留用户更明确的书签标题。
- 原始文件夹路径保存在来源元数据中。
- 导入来源和导入批次以标签或元数据形式保留。

## 十三、交互与体验要求

### 13.1 导入前

- 用户在执行前就能看到预估结果。
- 不允许“看不见后果就开始导入”。

### 13.2 导入中

- 必须展示实时进度。
- 导入任务可离开页面后继续。
- 如果是归档模式，需要展示当前归档条目与剩余数量。

### 13.3 导入后

- 成功、失败、跳过必须清晰分组。
- 失败原因尽量可理解，不只显示技术报错。
- 可对失败项批量重试。

### 13.4 空状态与提示文案

导入空状态建议强调：

- 可以先轻导入，再逐步归档。
- 支持浏览器书签和 URL 清单。
- 大批量导入建议优先使用“仅导入链接”。

## 十四、异常与边界条件

### 14.1 页面不适合归档

包括但不限于：

- 需要登录但当前浏览器无会话
- 页面已失效或 404
- 强交互型页面
- 浏览器内部页
- 文件下载链接

处理策略：

- 允许轻导入
- 不强制要求归档成功
- 在详情中标记“不可归档”或“归档失败”

### 14.2 大批量导入

当条目数超过阈值时：

- 采用分页或分片提交
- 优先完成轻导入
- 归档调度自动降速

### 14.3 重复运行导入

当用户多次导入同一批书签时：

- 默认应大部分命中去重
- 保留新的导入任务记录
- 不应生成大量重复书签

### 14.4 用户中断

如果用户关闭页面或刷新：

- 已创建的导入任务不得丢失
- 未完成的轻导入应可恢复或重新执行
- 已入队的归档任务应继续由扩展管理

## 十七、权限与技术约束

### 17.1 设计约束

- KeepPage 现有价值核心是 archive-first，不应为了批量导入而改成服务端抓取为主。
- 浏览器扩展是归档能力的首选执行环境。
- API 需要承接导入任务编排和结果状态，但不负责替代浏览器渲染环境。

### 17.2 数据约束

- URL 规范化逻辑必须与现有书签去重逻辑一致。
- 归档版本去重必须与现有 `htmlSha256` 逻辑一致。
- 导入记录需要可追溯，不得只保留最终书签结果。

## 十八、分阶段上线计划

### Phase 1：轻导入 MVP

目标：

- 先解决迁移门槛问题。

范围：

- 支持书签 HTML、URL 列表、CSV、TXT 导入
- 支持预览、去重、轻导入
- 支持导入历史和导入详情

不包含：

- 扩展原生浏览器书签读取
- 批量归档调度

### Phase 2：扩展联动归档

目标：

- 让导入条目可以转成完整网页归档。

范围：

- 扩展读取浏览器书签树
- 导入后排队归档
- 批量归档进度、失败重试、暂停恢复

### Phase 3：高级策略

目标：

- 提升规模化使用体验。

范围：

- 按文件夹、标签、域名补归档
- 按优先级归档
- 导入后智能建议
- 失败原因分类与优化建议

## 十九、开放问题

1. 文件夹结构在 KeepPage 中是否始终映射为真实文件夹，还是允许转为标签体系。
2. 导入后的标题策略是否允许后续被网页真实标题覆盖。
3. 归档模式下，默认并发度和节流策略是否按域名动态调整。
4. Safari 导出格式中的特殊字段是否单独兼容。
5. 导入记录在书签详情中展示到什么粒度。

---

## 二十、推荐结论

从产品和实现成本综合考虑，推荐采用以下路线：

1. 首期先做“轻导入 MVP”，尽快解决迁移门槛问题。
2. 在轻导入稳定后，再接入扩展侧“批量归档”能力。
3. 整个批量导入体系都围绕“先接收资产，再逐步归档”的思路展开。
4. 去重、版本和归档流程应最大化复用现有 KeepPage 基础能力，避免形成第二套收藏体系。

这套方案既能保持 KeepPage 的 archive-first 定位，也能显著降低新用户迁移成本。
